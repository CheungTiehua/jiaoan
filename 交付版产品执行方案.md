# LeKai 语文备课教案系统交付版执行方案

更新时间：2026-06-14

## 1. 当前共识

LeKai 不是大型 SaaS，也不是证据审计系统，而是一台面向约 20 位语文老师的校内/教研组内网备课盒子。

第一阶段的产品中心应当是：

> 帮语文老师更快做出一份可上课、可修改、可解释、可导出的教案。

证据链、bbox、高亮、DEE 风格 Evidence Object 仍然保留为底层能力，但不再作为老师日常使用的中心。它们主要用于：

- 给老师提供参考出处。
- 给教研组审核时证明不是模型凭空编造。
- 给校长、教研室、教育局展示系统具备规范化材料管理能力。

教学不是财务审计。语文教学允许教师自由发挥，也需要结合班情、学情、课堂生成进行调整。因此，系统不应把“每句话必须绑定证据”作为第一版核心目标。

## 2. 产品定位

### 2.1 服务对象

- 一线语文老师。
- 年轻教师。
- 教研组长。
- 校长或教务负责人。

### 2.2 使用场景

- 老师选择年级、册别、课文、课时，生成一份完整教案。
- 老师查看系统给出的设计说明，理解为什么这样设计。
- 老师参考教材、课标、教学指导、老教师案例，对教案进行调整。
- 老师导出 Word，用于上课、备课检查、教研交流。
- 教研组长查看材料覆盖情况和生成历史。

### 2.3 非目标

第一阶段不做以下事情：

- 不做大型多租户 SaaS。
- 不做复杂流程审批系统。
- 不强制每一句教案正文绑定 bbox。
- 不把老教师案例当作权威依据。
- 不把“考点分析、同行参考、辅导说明”做成四个并列的大产品。
- 不追求一步到位的完整 DEE 前端体验。

## 3. 功能降维方案

### 3.1 原功能问题

原先把系统拆成：

- 考点分析
- 同行参考
- 完整教案
- 辅导说明

这个设计对老师来说偏重。老师进入系统的主要目标不是分别消费四份材料，而是拿到一份可用教案，并理解这份教案怎么来的、怎么改。

### 3.2 新主界面

主界面只围绕一个动作：

> 选择课文 -> 生成教案

生成后页面结构：

```text
左侧或主区域：
  完整教案正文

右侧或辅助区域：
  设计说明
  参考材料
  规范依据

底部或顶部固定操作：
  导出 Word
  导出 Markdown
  保存历史
```

### 3.3 新功能分区

#### 生成教案

这是唯一主产物。

教案至少包含：

- 教材分析
- 学情分析
- 教学目标
- 教学重难点
- 教学准备
- 教学过程
- 板书设计
- 作业设计
- 教学反思建议

要求：

- 语言像真实老师写的教案。
- 结构完整。
- 可直接复制、修改、导出。
- 不因缺少材料而完全失败；缺资料时应明确提示“参考材料不足”。

#### 设计说明

解释这份教案为什么这样设计。

内容包括：

- 教学目标为什么这样定。
- 重难点为什么这样处理。
- 教学过程的主线是什么。
- 哪些环节适合年轻老师学习。
- 如果班级基础较弱或较强，可以怎么调整。

设计说明是“教老师怎么备课”的核心，不应写成行政汇报。

#### 参考材料

展示方法类材料：

- 《教学设计与指导》
- 老教师教案
- 本校案例
- 进修校案例
- 教研记录

文案统一使用：

- 参考：教学设计指导
- 参考：老教师案例
- 参考：本校案例

不使用“依据”“权威依据”等措辞。

#### 规范依据

展示规范类材料：

- 教材
- 课标
- 考纲
- 单元目标
- 考试说明

文案统一使用：

- 依据：教材
- 依据：课标
- 依据：考纲

第一阶段只要求展示材料名、课文/章节、摘录、页码。bbox 高亮可以保留，但不是核心。

## 4. 知识库策略

### 4.1 材料分层保留，但降低存在感

后端继续保留：

```text
normative: 规范依据
method_case: 方法参考
```

但前端不再强调复杂数据模型，只显示老师能理解的材料类型。

### 4.2 材料分类规则

规范依据：

- 教材
- 课标
- 考纲
- 单元目标
- 考试材料

方法参考：

- 教学设计与指导
- 老教师教案
- 本校案例
- 进修校案例
- 培训案例
- 教研记录

### 4.3 PDF 入库策略

不要求人工把整册 PDF 拆成每篇课文。

推荐流程：

1. 管理员上传整册 PDF 或 Markdown。
2. 系统 OCR 或解析全文。
3. 系统按页或按标题入库。
4. 管理员维护“课文 -> 页码范围/章节范围”的绑定。
5. 生成教案时优先检索当前课文绑定范围内的材料。

### 4.4 Markdown 入库策略

如果已有可读 Markdown，优先使用 Markdown。

原因：

- 入库快。
- 检索质量通常高于扫描 OCR。
- 适合先跑通交付版功能。

但 Markdown 缺少页码、bbox、原图定位，不能完全替代 PDF 的原文追溯能力。

## 5. 需要新增或调整的关键功能

### 5.1 课文材料绑定

这是下一阶段最重要功能。

目标：

```text
一年级 上册 教学设计与指导
  我是中国人 -> 第 1-8 页或对应 Markdown 章节
  我爱我们的祖国 -> 第 9-15 页或对应 Markdown 章节
  对韵歌 -> 对应章节
```

第一版可以做成简单管理表：

- 年级
- 册别
- 课文
- 材料
- 起始页
- 结束页
- 或 Markdown 标题范围
- 材料角色

### 5.2 教案生成界面重构

把四个大盒子收敛为一个主工作台：

- 顶部：选择年级、册别、课文、课时、补充要求。
- 中间：生成中的教案正文，流式输出。
- 右侧：设计说明、参考材料、规范依据。
- 操作区：保存、导出 Word、导出 Markdown。

### 5.3 Prompt 重写

Prompt 要围绕“完整教案”组织，不再让模型分别生成四份割裂内容。

新的生成逻辑：

1. 检索规范依据。
2. 检索方法参考。
3. 生成完整教案。
4. 同时生成简短设计说明。
5. 返回引用到的材料摘要。

### 5.4 导出重构

Word 导出应以教案为主体。

结构：

```text
完整教案

附录一：设计说明

附录二：规范依据

附录三：方法参考
```

附录不要喧宾夺主。

### 5.5 入库体验增强

第一阶段不必做复杂任务系统，但至少需要：

- 上传后显示正在解析。
- OCR 成功页数。
- 入库 chunk 数。
- 失败原因。
- 材料角色是否匹配。

### 5.6 管理端覆盖度简化

不做复杂证据健康评分。

只显示：

```text
课文        教材   课标/考纲   教学指导   老教师案例
对韵歌      有     有          有        无
小小的船    有     有          有        有
```

这个表对校内使用足够。

## 6. 执行阶段

### Phase 1：产品降维与主流程整理

目标：把系统从四功能并列改成“一个教案工作台”。

任务：

- 调整前端首页布局。
- 保留主按钮“生成教案”。
- 删除或折叠四个并列大盒子。
- 右侧改成“设计说明 / 参考材料 / 规范依据”。
- 生成结果默认先展示完整教案。
- 导出按钮放到明显位置。

验收标准：

- 老师进入页面后知道第一步要做什么。
- 生成结果首先是一份完整教案。
- 参考和依据不抢主内容。

### Phase 2：真实材料入库跑通

目标：先用 Markdown 和少量 PDF 把一个学期材料跑起来。

任务：

- 把已有 Markdown 按课文切分。
- 标注为 `teaching_guidance / method_case`。
- 教材、课标、考纲如果没有真实电子版，继续用占位材料。
- PDF 作为原件保留，必要时 OCR 入库。
- 建立课文与材料章节/页码绑定。

验收标准：

- 至少一个年级上册的主要课文能查到方法参考。
- 生成《对韵歌》《小小的船》《乌鸦喝水》等课文时，能命中对应材料。

### Phase 3：Prompt 与后端生成逻辑调整

目标：输出更像真实教案。

任务：

- 合并生成逻辑。
- 教案正文、设计说明、材料参考一次生成。
- 考点分析降级为教案中的“教学目标/评价关注点”。
- 同行参考降级为右侧“参考做法”。
- 辅导说明降级为“设计说明”。

验收标准：

- 流式输出几秒内开始。
- 教案结构完整。
- 设计说明能解释备课思路。
- 不把方法案例写成权威依据。

### Phase 4：导出与历史回看

目标：老师能拿走成果。

任务：

- Word 导出排版优化。
- Markdown 导出保留。
- 历史记录打开后仍能看到教案、设计说明、依据、参考。
- 附录分清“规范依据”和“方法参考”。

验收标准：

- 导出的 Word 能直接修改。
- 教案主体在前，附录在后。
- 历史记录可再次导出。

### Phase 5：交付前回归

目标：确认能给 20 位老师试用。

任务：

- 登录、生成、保存、历史、导出回归。
- 管理员上传材料回归。
- PDF OCR 入库回归。
- Markdown 入库回归。
- 备份恢复回归。
- DeepSeek 配置回归。
- 弱网或模型失败时错误提示回归。

验收标准：

- 老师能完成一次完整备课。
- 管理员能完成一次材料入库。
- 系统失败时给出可理解提示。
- 不出现“生成不了任何东西”的阻断问题。

## 7. 当前项目状态判断

### 已经完成或基本完成

- DeepSeek 模型配置调整。
- 流式生成基础能力。
- 规范依据/方法参考的后端分层。
- 占位知识库。
- PDF OCR 入库基础能力。
- Markdown / docx / txt 入库基础能力。
- evidence 元数据保留。
- Word / Markdown 导出基础能力。
- 前端代理后端 API。

### 仍然不足

- 产品界面仍偏复杂。
- 四个功能并列的心智模型需要改。
- 整册 PDF/Markdown 到课文的绑定还没完成。
- Prompt 仍需要按“交付教案”重写。
- 入库状态提示不够产品化。
- 真实材料尚未系统入库。
- 端到端验收还不够。

## 8. 交付优先级

最高优先级：

1. 首页工作台降维。
2. 课文材料绑定。
3. Markdown 按课文切分入库。
4. 教案生成 Prompt 重写。
5. Word 导出稳定。

中优先级：

1. PDF OCR 全量入库。
2. 入库进度和日志。
3. 覆盖度简表。
4. 历史详情优化。

低优先级：

1. bbox 高亮完整体验。
2. 严格 citation validation。
3. 复杂 evidence pack。
4. 多角色复杂审批。
5. 大型 SaaS 化能力。

## 9. 下次执行建议

下次不要继续散修 bug。建议按以下顺序推进：

1. 先改前端首页，把产品形态收敛成“教案工作台”。
2. 写 Markdown 切分脚本，把《一年级上册-语文教学设计与指导.md》按课文切成 method_case。
3. 增加课文材料绑定模型。
4. 调整生成接口，让它返回：
   - lesson_plan
   - design_notes
   - normative_sources
   - method_references
5. 重写导出。
6. 用一年级上册选 3 篇课做完整验收。

建议首批验收课文：

- 我是中国人
- 对韵歌
- 小小的船
- 乌鸦喝水

这几篇覆盖入学教育、识字、阅读、童话/故事类文本，足够暴露系统问题。

## 10. 最终交付定义

第一版交付不是“证据链系统完成”，而是：

> 一个语文老师能在 3 到 5 分钟内生成一份结构完整、语言自然、可修改、可导出的教案，并能看到这份教案参考了哪些材料、为什么这样设计。

达到这个标准，就可以进入校内试用。
